O Java, uma coisa muito positiva dele é que ele é o padrão para a programação orientada a objetos. Quando você vai construir uma aplicação muito robusta, uma das coisas que você tem que levar em consideração é que você precisa ter entidades dentro do teu projeto que sejam capazes de realizar tarefas simultâneas sem necessariamente rodarem o código. O que isso significa? Você precisa ter um objeto dentro do teu projeto chamado Security Resources. Ele não roda nada, mas ele garante com que exista uma consistência entre as classes. E se alguma coisa não estiver consistente, ele vai te avisar. Tem alguma coisa errada aqui. Você precisa ver isso daqui. Então, em termos de escalabilidade, isso ajuda muito para você não perder... Para você não ter problemas exponenciais de desenvolvimento de código. Essa é a primeira coisa. Outra coisa também com relação à programação orientada a objetos, é muito mais fácil para você desenvolver aplicações orientadas a objetos em relação à realidade. Ou seja, observar o que é aquilo é muito mais fácil, muito mais tangível. Nós usamos o modelo MVC, ou seja, Model View Controller, onde o model faz a correlação, ele faz a gestão dos dados, o view é o que o usuário vê e o controller é a gestão entre os dados e a UI, no caso, que o usuário tem acesso. Dentro desse sistema, você precisa saber várias coisas, você precisa saber projeto Maven, que é para você fazer a gestão das dependências das bibliotecas que não são naturais do Java. Ou seja, o Java tem um certo toolkit que você tem e depois você tem várias outras tecnologias conjuntas, várias bibliotecas que você vai puxando. E aí você precisa ter esse controle certinho de dependências, os pacotes que a gente chama. E, ao mesmo tempo, uma coisa positiva da orientação a objetos em Java, pelo menos, é que apesar de criar aberrações como nomes de classes gigantescos, ele te permite você fazer o modelo MVC, Model View Control, de um jeito muito eficiente, muito fácil, muito simples.
